## Executive Summary

The spec is unusually disciplined — 23 requirements, 19 acceptance scenarios, and success
criteria written to be falsifiable rather than merely passable. The plan is grounded in
nine findings produced by actually running graphify 0.9.9, which already caught the
design-breaking fact that no `graphify build` subcommand exists. That research habit is
the plan's main strength and should not be diluted.

Four must-address findings remain, and three of them share one root cause: **the research
stopped one probe short**. It verified the happy path (`graphify update .` from the
project root) and generalised from it. Two follow-up probes run during this critique
falsify that generalisation — `--full` does not exist as a flag, and running with a path
different from the working directory writes into *two* directories. Both are load-bearing
for requirements the spec already commits to.

The fourth is a scope drift: the plan introduces a code/non-code split that the spec never
mentions, and for a tool aimed at Spec Kit projects — whose most valuable content is
Markdown — that split silently narrows the feature's value without anyone having decided
to narrow it.

None of this is architectural. The delegation boundary, the placement of confirmation in
the agent layer, and the refusal to vendor graphify all hold up under pressure. The fixes
are corrections to specific claims, not a redesign.

## Engineering Lens Findings 🔬

### Architecture Soundness

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E1 | 🎯 | **`--full` does not exist.** The plan's `BuildRequest.mode`, the command contract's `--full` argument, User Story 2 scenario 3 ("requests a full rebuild… the whole project is re-examined"), and SC-003's ratio all depend on a full-rebuild mode. Probed directly: `graphify update . --full` → `error: unknown update option: --full`. `--force` is **not** an equivalent — with `--force` the tool still reported `No code-graph topology changes detected; outputs left untouched.` | A full rebuild is achievable by removing `graphify-out/graph.json` before invoking `update` — verified: after `rm graph.json`, the next `update` rebuilt from scratch. Record this in `research.md` as its own finding, and specify it in the script contract, including that deleting a tool-owned file is a deliberate, documented exception to "never write into `graphify-out/`". That exception needs stating precisely, because it sits one inch from Principle XVII. |
| E2 | 💡 | `BuildOutcome` enumerates nine values; SC-005 enumerates seven outcomes. The design outgrew the criterion, so a test written against SC-005 would pass while leaving `interrupted-state` and `failed` unchecked. | Update SC-005 to the full set, or state which outcomes are out of scope for v1 and why. |
| E3 | 💡 | The data model gives `declined` an exit code, but `declined` is decided in the agent layer and never produced by a script. Two artifacts describe the same enum with different owners. | Split the enumeration: outcomes the script emits, and outcomes the command emits. One table pretending to be both is how an implementer ends up adding dead code for an exit path that cannot occur. |

### Failure Mode Analysis

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E4 | 🎯 | **Output lands in two directories when the scope root is not the working directory.** Probed: from `elsewhere/`, running `graphify update ../proj` wrote the graph to `proj/graphify-out/` *and* created `elsewhere/graphify-out/manifest.json`. Since `scope.root` is configurable, a maintainer with a non-default root gets a stray `graphify-out/` in their project root — one that `.gitignore` happens to cover here, but which violates FR-015's promise about where writes go, and which the report would not mention. | Require the script to change directory to the resolved scope root before invoking the tool, and assert afterwards that no `graphify-out/` was created anywhere else. Add a quickstart scenario with `scope.root` set to a subdirectory — the current scenarios all use the default and would never catch this. |
| E5 | 🎯 | FR-019 (interrupted build) and FR-020 (concurrent build) are specified as behaviour with no mechanism, and no research backs either. Exit codes 6 and 7 are asserted in the contract but nothing establishes how the conditions are detected. Worse, the obvious place to put a lock is inside `graphify-out/` — which is tool-owned, making the natural implementation a principle violation. | Put both sentinels in the extension-owned directory, never in `graphify-out/`. For interrupted state, a plausible detector is `manifest.json` present with `graph.json` absent — that exact combination was produced during probing and is worth verifying deliberately before relying on it. For the lock, use an atomically-created directory, and specify the stale-lock policy: a lock whose owning process is gone must be reclaimable, or the first crash makes the command permanently unusable. |

## Resolution (2026-07-22, same day)

All four must-address findings and every actionable recommendation were applied. The three
open questions were answered by the maintainer and by measurement:

| Question | Answer | Where it landed |
|---|---|---|
| **P1** — who is this for? | A publicly maintained Spec Kit plugin, installed by strangers into projects the author will never see | Spec "Audience" paragraph; plan "Distribution" section. It raised the stakes on the version ceiling, the distribution matrix, and the outcome vocabulary — all three exist because the reader cannot ask a follow-up question |
| **P2** — is code-only enough for v1? | Yes, but only with the handoff made first-class | New FR-013b, User Story 1 scenario 9, SC-009, and step 6 of the command contract. A stranger arrives expecting the graph the upstream tool advertises; getting the code half is acceptable when the report says so every time, and unacceptable silently |
| **E9** — does graph size threaten the counting pass? | No. 50,000 nodes / 200,000 links (45.7 MB) counted in 0.65 s | Research R15. Adding a streaming parser would have been complexity with no subject |

One finding grew worse under investigation. E7 asked whether exclusion honouring had been
verified; the probe found that `vendor/v.py` entered the graph and that the tool exposes no
exclusion option at all. `scope.exclude` was removed from v1 rather than kept as a setting
that silently does nothing — see research R13.

# Staff Review: Graph Build Command

**Date**: 2026-07-22
**Feature**: [spec.md](../spec.md) · [plan.md](../plan.md) · [tasks.md](../tasks.md)
**Scope**: `git diff main` — 32 files, +4701 / −7
**Verdict**: ❌ **CHANGES REQUIRED**

---

## Executive Summary

The implementation follows its contracts closely, and the parts the feature exists for —
the failure paths — are genuinely exercised rather than assumed. The test suites assert
both exit code and reported outcome, were themselves mutation-tested, and two research
findings (R16, R17) corrected the specification during implementation instead of being
worked around. That is the right direction of correction.

One blocker was found, and it is the kind that a green suite hides rather than catches:
**the bash script leaks its lock when the scope root is not the working directory**. The
lock path is relative, the script `cd`s into the scope root after acquiring it, and the
cleanup then resolves the relative path against the wrong directory. The lock is never
released.

The suite passed anyway, for an instructive reason: stale-lock reclamation makes a leaked
lock invisible on the next run. A safety mechanism masked the defect that the same test
was written to catch. This is exactly the shape Principle XV warns about — the concurrency
assertion has only ever been observed passing against a lock it created itself, never
against one the script leaked.

The same bug does not exist in the PowerShell variant, which pops its location before
releasing. So the finding is also a **parity divergence**, and parity was claimed on the
basis of a suite that runs both variants — but through a scenario blind to this difference.

## Test Quality Assessment

**Strong.** The suite asserts exit code *and* outcome, which is the difference between
"something failed" and "the right thing failed". Failure states are constructed
deterministically — an empty `PATH` for absence, a stub binary for a wrong version, a
deleted `graph.json` for interruption — so nothing depends on timing. The mutation check
(reading `edges` instead of `links` fails 5 assertions; reporting an empty scope as success
fails 1) is real evidence the assertions bite.

**The gap** is F2, and it is a specific kind worth naming: an assertion that passes both
when the code is right and when it is wrong. The concurrency test creates its own lock, so
it never observes whether the script releases one. Every other failure path in this suite
is constructed *and then verified as absent afterwards*; this one is only constructed.

## Resolution (same session)

F1 and F2 were fixed immediately, in the order the review asked for:

1. The lock-release assertion was added to `scripts/test-graph-build.sh` **first**, and
   confirmed failing against the unfixed script — `37 passed, 1 failed`. A regression test
   that never saw the bug is not a regression test.
2. `EXT_DIR` and `LOCK_DIR` are now resolved to absolute paths in `main()`, before anything
   can change directory. Bash suite: **38 passed, 0 failed**.
3. The same assertion was added to `scripts/test-graph-build.ps1`, so the divergence in F3
   cannot reappear in either direction. PowerShell suite: **26 passed, 0 failed**.

F4–F8 remain open and are tracked as follow-ups. None of them leaves state in a user's
project, which is what separated F1 from the rest.

**Post-fix verdict**: ⚠️ **APPROVED WITH CONDITIONS** — the conditions being F4 (repeated
graph parsing) and F5 (project-root resolution), neither blocking.

## Verdict (as found)

❌ **CHANGES REQUIRED**

F1 violates FR-015 and leaves state behind in a user's project. F2 is the reason it reached
review at all, and fixing the bug without fixing the test would leave the same class of
defect free to return.

Neither is deep. Both are one-line-scale changes. What they cost is the claim that this
feature was verified — which is recoverable by making the assertion fail first, then pass.
